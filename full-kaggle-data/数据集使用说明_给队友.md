旧数据集测试集只有 1,196 条、且没有平衡采样机制；新数据集：

## 三、关键列说明（`02_train_full_weighted.csv`）

| 列名 | 作用 | 怎么用 |
|---|---|---|
| `filename` | OGG 相对路径，如 `abethr1/XC128013.ogg` | 拼接音频根目录读 wav |
| `primary_label` | 鸟种代码（1,126 类） | 分类目标 |
| `sampler_weight` | **平衡采样权重** = 1 / 该物种训练样本数 | 传给 `WeightedRandomSampler`（PyTorch）或无需使用（GBDT） |
| `loss_class_weight` | **温和类别加权** = `clip(1/sqrt(count) / 均值, 0.25, 4.0)` | 传给分类 loss 的 `weight=` |
| `cv_fold` | 1–5，每行属于哪个验证折 | 也可不读此列，直接用单独的 `cv_fold*_train/val.csv` |
| `snr_tier` | High / Medium / Heavy | 评估时按档位报告，不作训练特征 |
| `class_count_train` | 该物种在训练池的样本数 | 参考，不作特征 |

> ⚠️ **不要把 `sampler_weight` 和 `loss_class_weight` 相乘！** 前者管"抽到频率"，后者管"错一次扣几分"。

## 四、稀有物种音频增强

1,126 种鸟类中，有 **159 种**训练样本不足 15 条。对这些低资源物种做增强，每类补到至少 15 条。

### 4.1 增强方法（共用，4 种）

三个算法共用同一套增强底层（`audio_augmentation.py`），保证可比：

| 方法 | 参数 | 模拟的现实变化 |
|---|---|---|
| 时间拉伸 | 0.85x / 0.90x / 0.95x / 1.05x / 1.10x | 鸣叫节奏变化 |
| 音高偏移 | ±1 / ±2 半音 | 个体/地域音高差异 |
| 加高斯噪声 | 5 / 10 / 15 dB SNR | 不同录音环境 |
| 音量缩放 | 0.70x / 0.85x / 1.15x / 1.30x | 距离和设备增益差异 |

**规则**：每条原始音频最多 16 个变体，每物种最多新增 50 条，目标每类至少 15 条。增强发生在**波形阶段**，只在每折的训练集上做，验证集和测试集不增强。

### 4.2 需要上传的文件

Kaggle 上跑需要在 `src/` 目录放两个 `.py` 文件：

| 文件 | 说明 | 谁提供 |
|---|---|---|
| `audio_augmentation.py` | 共用底层，4 种增强方法 | YAMNet 队友已上传，大家用同一份 |
| `augmentation_glue.py` | 各算法调用增强的胶水函数 | 各自一份（见下表） |

| 算法 | 胶水文件位置 | 核心函数 |
|---|---|---|
| YAMNet | 已集成，无需上传 | 已调用 |
| LightGBM | `bird-vocalization-classifier/lightgbm/augmentation_glue.py` | `augment_for_lightgbm()` |
| FastAI | `bird-vocalization-classifier/fastaicode/augmentation_glue.py` | `augment_for_fastai()` |

### 4.3 接入要求

增强在 5 折循环里每折自动触发一次，跑完会打印类似：

```
[增强] 低资源物种 159 种 (少于 15 条)
[增强] 新增增强样本 1144 条
```

看到这个就说明接入成功。YAMNet 已经集成好了，LightGBM 和 FastAI 的接入要求见第五节的各模型要求。

## 五、三个模型需要做到的事

### YAMNet ✅ 已就绪

`YAMNet/src/e2e/yamnet_e2e_all_in_one.ipynb` 已使用 `03_test_holdout.csv` / `02_train_full_weighted.csv` / `class_weights.csv` / `cv_fold{N}_train/val.csv`，并已读取 `loss_class_weight` 注入 loss。增强也已集成。**无需改动，直接在 Kaggle 跑。**

### LightGBM ❌ 需要自检

请确保你的最新代码做到：

1. **读对文件**：测试集用 `03_test_holdout.csv`，5 折用 `cv_fold{N}_train/val.csv`（不要再找 `ml_*` 前缀的文件）。
2. **类别数对齐**：标签空间以 `02_train_full_weighted.csv` 的 **1,126 类**为准，`label2idx` 重新构建。
3. **接入类别加权**：`class_weights.csv` 的 `loss_class_weight` 按 `primary_label` 映射后，作为 `sample_weight=` 传给 `lgb.train`。LightGBM 是 GBDT，不用 `WeightedRandomSampler`，`sampler_weight` 那列可忽略。
4. **接入增强**：在每折的 `split_xy` 之后、`lgb.train` 之前，调用 `augment_for_lightgbm()` 并把增强后的特征拼接到 `X_tr`。种子用 `SEED + fold`，保证每折可复现。

### FastAI ❌ 需要自检

请确保你的最新代码做到：

1. **读对文件**：测试集用 `03_test_holdout.csv`，5 折用 `cv_fold{N}_train/val.csv`。
2. **类别数对齐**：标签空间以 `02_train_full_weighted.csv` 的 **1,126 类**为准。
3. **接入两个权重**（FastAI 是深度学习，两个都要用）：
   - `sampler_weight` → `WeightedRandomSampler`（控制每类进 batch 的频率）
   - `loss_class_weight` → `CrossEntropyLoss(weight=...)`（控制每类错误对 loss 的影响）
   - 两者不可互换，也不可相乘。
4. **接入增强**：在每折的训练集转频谱图之后、验证集转频谱图之前，调用 `augment_for_fastai()`。增强的频谱图 PNG 文件名格式为 `{stem}_aug{N}_{species}.png`，存在 `Config.IMG_DIR` 下。种子用 `42 + fold`。验证集不做增强。

## 六、评估规范（三个模型统一）

最终评估前**不要**读 `03_test_holdout.csv`，只用 CV 选模型。选完模型后统一在 `03_test_holdout.csv` 上跑：

1. **clean**（原始音频）→ 报告 Top-1、macro-F1、balanced accuracy，并按 `snr_tier` 分档报告；
2. **加噪**：固定种子 42 叠高斯白噪声，SNR 档位 **5 dB / 0 dB / -5 dB**；
   - 公式：`SNR_dB = 10*log10(P_signal/P_noise)`，叠后峰值归一化到 [-1,1]；
3. **推理速度**：预热 5 次 + 测 50 次，输出 ms/条，记录显存（LightGBM 走 CPU 记 N/A）。

三个模型必须用同一套测试集和同一套噪声种子，结果才公平可比。

## 七、跑的顺序与常见问题

每个人各自在 Kaggle 跑自己的 notebook，顺序不变：

```
Cell 1: 训练（含增强，每折自动触发，不需要单独 cell）
Cell 2: 噪声鲁棒性评估
Cell 3: 推理速度测量
```

**Q: 为什么不直接生成增强文件到磁盘，大家共用？**

A: 两个原因：1) 在线增强每次 epoch 可以对同一条音频随机选不同方法，等价于无限增强空间，离线生成是固定的；2) 离线生成后很难保证增强样本不泄漏进同折验证集。每折在训练集上调一次最安全。

**Q: 增强后训练集变大，会不会拖慢训练？**

A: 只对 159 个低资源物种增强，新增约 1,144 条，相对 12.9 万总量只增加不到 1%，基本不影响速度。

**Q: 增强 import 的路径写什么？**

A: 看 `src/` 目录在 Kaggle 上实际挂载在哪。常见是 `/kaggle/working/src` 或 `/kaggle/input/<dataset>/src`。上传后看一眼路径填进去。
